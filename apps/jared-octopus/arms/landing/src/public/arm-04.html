Data &Memory.

Arm 04 — Nothing is forgotten. Persists every observation, every power reading, every agent decision trace. The system's long-term memory and the head's context source.

04

What it does

Structured metadata and raw signal storage

Arm 04 operates a two-tier storage system. Structured observation metadata — frequency, timestamp, antenna ID, duration, noise floor, candidate signals, power state, calibration version — lives in a SQLite database with full-text search and indexed queries. Raw IQ sample files live on disk as GZIP-compressed binary blobs, referenced by their observation ID in the database.

The separation matters. The head can query "show me all 1420 MHz observations from the last 7 days where the noise floor was below -100 dBm" without touching a single IQ file. IQ files are only loaded when arm 02 needs to re-process historical data — which is rare and deliberately slow to discourage doing it during active sessions.

The RFI mask and environmental baseline

Arm 04 is the custodian of the RFI mask — the frequency-domain map of known interference sources that arm 02 uses for threshold detection. The mask is updated after every observation session: arm 02 reports which bins were persistently elevated, and arm 04 merges them into the mask with a confidence score. Bins with high confidence (seen in >80% of recent sessions) are hard-masked; bins with lower confidence are soft-masked and still reported but down-weighted.

The calibration correction table from arm 05 is also stored here, versioned by timestamp. Each observation record references the calibration version that was active when it was captured, so historical data can be re-processed with updated corrections.

Query interface for the head

Before the head dispatches an observation directive, it typically queries arm 04 to understand what's been seen at that frequency before — noise floor history, previous candidates, RFI mask status, and the last calibration timestamp. This context shapes the head's decisions: if the noise floor has been creeping up over the last week, the head might request a calibration run before committing to a long observation.

The observatory's long-term memory. Arm 04 is what makes the system learn — every session informs the next.

Status

  • State: Active
  • Database: SQLite 3.x
  • IQ storage: GZIP-compressed binary
  • Default retention: 30 days rolling
  • Compression ratio: ~3:1 on typical IQ data
  • Energy draw: <5W (disk I/O only)

Storage layout

Schema.

observations table

  • id — UUID, primary key
  • timestamp_ms — Unix ms, indexed
  • freq_mhz, bw_mhz — observation parameters
  • antenna_id — from arm 01 manifest
  • duration_sec, samples_captured
  • noise_floor_dbm — from arm 02
  • cal_version — FK to calibrations table
  • iq_path — relative path to raw file (nullable)
  • flags — JSON: partial, cal_stale, rfi_events

candidates table

  • id, observation_id — FK to observations
  • freq_mhz, bandwidth_khz
  • peak_dbm, snr_db
  • confidence — 0.0–1.0 from arm 02
  • reviewed — bool, set by head after inspection

rfi_mask table

  • freq_mhz — centre of masked bin
  • bandwidth_khz — width of mask
  • confidence — 0.0–1.0
  • first_seen_ms, last_seen_ms
  • occurrence_count

File layout

  • data/observations/ — SQLite DB file
  • data/iq/YYYY-MM/ — monthly IQ directories
  • data/iq/YYYY-MM/<obs-id>.iq.gz — raw files
  • data/cal/ — calibration tables (JSON)
  • data/rfi/rfi-mask.json — current mask snapshot

MCP Tools

The interface.

log(observation: Observation)

Persist a completed observation record. Called by arm 02 (via head) at the end of every capture session. Writes metadata to SQLite and, if IQ data is attached, streams the compressed file to disk.

Returns: { id: string, stored_bytes: int, iq_path: string|null }
Example: log({ freq_mhz: 1420.405, duration_sec: 60, ... })
query(filter: QueryFilter, limit: int = 50)

Query the observation database. Filter supports frequency range, time range, minimum SNR, antenna ID, and flag conditions. Used by the head before scheduling observations and by arm 06 when preparing uplink summaries.

Returns: Observation[] (metadata only, no IQ)
Example: query({ freq_min: 1420, freq_max: 1421, since_ms: Date.now()-86400000 })
get_rfi_mask(freq_min_mhz: float, freq_max_mhz: float)

Return the current RFI mask entries within the given frequency range. Called by arm 02 on startup and after each observation session to load the latest mask state.

Returns: RFIMaskEntry[]
Example: get_rfi_mask(1400.0, 1450.0)
update_rfi_mask(entries: RFIMaskEntry[])

Merge new RFI observations into the mask. Called by arm 02 after each session. Arm 04 applies a confidence decay function to existing entries — bins not seen in recent sessions lose confidence and eventually get retired.

Returns: { updated: int, added: int, retired: int }
Example: update_rfi_mask([{ freq_mhz: 1430.5, bandwidth_khz: 200, confidence: 0.9 }])
get_context(freq_mhz: float, lookback_days: int = 7)

Return a structured context summary for a given frequency — recent noise floor history, candidate signal count, RFI mask status, and last calibration age. This is what the head reads before scheduling an observation.

Returns: { noise_floor_history: float[], candidate_count: int, rfi_masked: bool, cal_age_hours: float }
prune(older_than_days: int = 30)

Remove IQ files and observation records older than the retention threshold. Metadata-only records (no IQ file) are retained indefinitely. Called by a daily cron job triggered by the head.

Returns: { records_removed: int, bytes_freed: int }
Example: prune(30)

Failure modes

What breaks.
How it recovers.

disk full

Storage fills before scheduled pruning

Arm 04 monitors available disk space and pre-emptively prunes when free space drops below 10%. If that's not enough, it stops accepting raw IQ files (metadata-only mode) and alerts the head. Observations continue but without raw data archival.

DB corruption

SQLite integrity check fails

Arm 04 attempts a VACUUM and integrity repair. If the database is unrecoverable, it renames the corrupted file, creates a fresh database, and alerts the head. Historical data is lost but current operations continue. IQ files on disk are unaffected.

write failure

IQ file write fails mid-stream

The partial file is deleted and the observation is logged with an iq_failed flag. Metadata is preserved. Arm 04 reports the failure to the head with disk health context (SMART status if available).

Integration

How it fits.

Integration contract

Receives from
Head: log requests (with observation data from arm 02), query requests, prune commands, RFI mask updates.
Reports to
Head: query results, context summaries, storage health, fault events.
Loaded by
Arm 02 (RFI mask, calibration table) and arm 05 (calibration history) — loaded at startup via head-mediated fetch, not direct arm-to-arm calls.
Produces
Persistent observation archive. Query interface. RFI mask. Calibration version history. Context summaries for head scheduling.
