Calibration.

Arm 05 — Science needs ground truth. Measures the system's own noise floor, gain response, and thermal drift so arm 02's observations mean something.

05

What it does

Noise figure and gain characterisation

Before the system can meaningfully measure signal power, it needs to know its own response. Arm 05 runs periodic characterisation sweeps using a noise source — typically a switched avalanche diode noise source connected to the antenna port via an RF relay. With the noise source on and off, the Y-factor method gives the receiver noise figure across the frequency range. The result is a gain-vs-frequency correction table stored in arm 04 and applied by arm 02 to convert raw ADC counts to calibrated dBm values.

The sweep runs in under 5 minutes and is scheduled by the head at session start, after significant ambient temperature changes (>5°C from last calibration), and on arm 02's explicit request when it detects a baseline drift anomaly.

Thermal drift correction

SDR hardware — especially RTL-SDR dongles — are notoriously temperature-sensitive. The gain of the tuner changes measurably as the silicon warms up over the first 20–30 minutes of operation. Arm 05 characterises this warm-up curve during the initial calibration session and builds a temperature-indexed correction model. At runtime, arm 02 applies the correction based on the current thermal reading from arm 03's DS18B20 sensors mounted near the SDR.

For long observation sessions, arm 05 can be tasked to run a brief recalibration check mid-session — a 30-second noise source sweep — to verify the model is still accurate. If it has drifted by more than 0.5 dB, it issues an updated correction table to arm 02 and logs the event to arm 04.

End-to-end signal path verification

Arm 05 also verifies the complete signal path: noise source → LNA → coax → SMA connector → SDR. It can inject a known tone (using the HackRF's transmit capability if available) and compare the received power against the expected value. Discrepancies beyond tolerance indicate a fault in the signal chain — a loose connector, a failing LNA, or a degraded coax run — and are reported as signal path faults to the head.

The system's internal standard. Without calibration, arm 02 is just counting noise. With it, it's doing science.

Status

  • State: Standby (scheduled)
  • Method: Y-factor noise figure
  • Noise source: Switched avalanche diode
  • Frequency range: Matches arm 02 SDR
  • Cal duration: ~5 min full sweep, ~30s spot check
  • Energy draw: 8–15W (noise source on)

MCP Tools

The interface.

calibrate(mode: string = "full")

Run a calibration session. Mode is "full" (complete gain sweep, ~5min), "spot" (single-frequency noise floor check, ~30s), or "thermal" (temperature drift correction only, ~60s). Returns a new correction table version.

Returns: { version: string, mode, gain_table: [{freq_mhz, gain_db, noise_figure_db}], timestamp_ms: int }
Example: calibrate("full")
get_correction_table(version: string = "latest")

Retrieve a calibration correction table by version. Called by arm 02 at startup and after any calibration event. Tables are stored in arm 04 and retrieved via this tool.

Returns: { version, timestamp_ms, entries: [{freq_mhz, gain_correction_db, noise_floor_dbm}] }
Example: get_correction_table()
verify_signal_path()

Inject a known tone and verify end-to-end signal path integrity. Uses HackRF transmit if available; falls back to noise source comparison. Returns pass/fail and the measured vs expected power delta.

Returns: { ok: bool, delta_db: float, fault_location: string|null }
Example: verify_signal_path()
get_cal_status()

Return the age and quality of the current calibration. The head uses this to decide whether to request a recalibration before a long observation session.

Returns: { current_version, age_hours: float, temp_delta_c: float, drift_db: float, needs_recal: bool }

Failure modes

What breaks.
How it recovers.

noise source fail

RF relay or noise source doesn't respond

Arm 05 reports calibration unavailable and flags all subsequent arm 02 observations with cal_unavailable. Science continues but with reduced confidence. The head is alerted to request manual inspection.

thermal drift

Measured gain deviates from model by more than 1 dB

Arm 05 flags the current correction table as stale and generates a new one. If the new calibration produces consistent results, the updated table is pushed to arm 04 and arm 02 loads it. If calibration produces inconsistent results (variance >0.3 dB between runs), it reports a hardware instability fault.

path fault

Signal path verification finds unexpected insertion loss

Arm 05 reports the fault location estimate (based on which frequency ranges are affected) and the magnitude of the discrepancy. The head pauses science operations and alerts the operator for physical inspection.

Integration

How it fits.

Integration contract

Receives from
Head: calibration run requests, status queries, recalibration triggers from arm 02 drift alerts.
Reports to
Head: calibration results, correction tables (forwarded to arm 04), drift alerts, signal path faults.
Context needed
Temperature readings from arm 03 (fetched via head at calibration time). Previous calibration versions from arm 04.
Produces
Versioned gain correction tables. Noise figure profiles. Signal path health reports. Thermal drift model updates.
Scheduling
Runs at session start, on 5°C+ thermal delta, on arm 02 drift alert, and on operator request. Never interrupts an active arm 02 observation.
