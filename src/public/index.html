Origin

"If you were embodied as a submarine with arms, how many arms would you like to use? Consider the same energy budget. How would you integrate those arms into the OpenClaw framework — CLI, MCP, or API?"
— Maksym Huczynski · Submarine design session · March 2026

Six arms.

An octopus has eight but two are for locomotion. A fixed desk tool doesn't move — it works. Six arms at 60° radial spacing fits a 200W bench supply (~150W peak), and a single head can meaningfully supervise six agents before coordination overhead kills the gain. Eight starts losing efficiency.

MCP. Not CLI or API.

CLI forks a new process per command — 150ms+ startup kills real-time sensor feedback. REST polling wastes energy and drops state between calls. MCP is persistent, bidirectional, streaming. State lives in the server. Arms and head stay connected continuously. Purpose-built for agents talking to agents.

Why MCP

CLI
Insufficient.

Fork overhead per command. 150ms+ startup. No streaming. Breaks under continuous sensor feedback.

REST API
Partial.

Request and response only. Polling wastes energy. State lost between calls. Not built for agents.

MCP
Persistent.

Bidirectional socket. Streaming sensor data. State lives in the server. Purpose-built for agents talking to agents.

OpenClaw topology

Head as operator.
Arms as agents.

The head doesn't do science — it decomposes tasks and arbitrates conflicts. If arm-01 wants to swap antennas while arm-02 is mid-observation, the head decides. Arms never message each other. Everything routes through the head. The graph stays acyclic.

Each arm is "Jared" — same base model, different MCP toolset, different memory namespace. Swap in a better SDR without touching the head. Upgrade the operator model without rewiring the arms.
